Problem Statement

Traditional evaluation methods rely on a final exam or assessment to measure a student's growth. However, exam scores often give noisy, inaccurate results, especially for students who aren't good test-takers.

In our project, we explore the ways a studentâ€™s process can be used as a better measurement of their growth in ability. To examine these effects, we will use student submissions in the Hour of Code task on Code.org, and try to evaluate students using modern ML techniques.

Data Analysis

Zipf Distribution

The main reason assessing student code is difficult is because student code is Zipf distributed, which means that it is hard to collect a wide enough dataset to capture enough student submissions in order to use usual deep learning techniques. Thus, a more effective approach is to generate our own data using a recursive grammar.

Data Visualization

The first major deliverable for our project was creating a command-line interface for inspecting and visualizing student data.

We wanted this tool to be fast and easy to use, enabling us to skim through hundreds of submissions quickly and get a good sense of the data we were working with. We wrote the tool in Python and allowed the user step through different students, each with different submissions, for all 10 problems using keyboard shortcuts.

The interface contains options to view code in the autoformatted form or in the abstract syntax tree form, turn on/off our predictive algorithm, and show/hide the student's final report card.

Our goal for this data visualization tool is to allow future research on this dataset to more easily examine specific data examples, in order to qualitatively measure a model's predictive ability.

Grammars

Generating data samples using a grammar is a novel technique in zero-shot learning, which allows us to use human knowledge to encode labels into specific classes of examples. The goal of our grammar is to match the distribution of the true distribution of student submissions.

Generating + Verifying Grammars

In our third assignment, we explored how to use Chris's idea-to-text engine with our grammars to randomly generate student code with rubric labels.

However, since we were going to be generating our data from a grammar for each of the problems we were interested in, we needed some way of evaluating whether our grammars were complete enough, and sufficiently approximated the true data's distribution.

To do so, we wrote grammars for problems 1, 2, 3, and 4. We then evaluated our grammars using completeness as a primary metric, and then L2 distance as a secondary metric. While we did not get as high accuracy on our grammars for problems 3 or 4 compared to problem 1, we still did fairly well given the Zipf distribution of the data.

Estimator

To estimate student ability, we used the following process:

  • Create grammars for multiple Code.org problems.
  • Validate our grammars by comparing them to the sample data.
  • Train an LSTM using each grammar.
  • Collect rubric predictions for all student submissions.
  • Identify features of evolving rubric items.
  • Report a score categorizing a student's learning and compare different students.

Transition Matrix

We use a transition matrix to help assign a transition learning score for each rubric item from one submission to its subsequent submission. For an (i, j) entry, i refers to whether or not the first submission contains the current rubric item and j refers to whether or not the second submission contains the current rubric item. 0 denotes not containing a rubric item and 1 denotes containing a rubric item.

Therefore, we assign a transition score of +1 (applied to a discount factor which we shall in the following section) when a student goes from 1 to 0 (i.e. having a label to not having a label, yay!) and a transition score of -0.5 when a student goes from 0 to 1 (this allows a student to be able to recover by going from 1 to 0 and gaining a learning transition score of 1 discounted by a factor). If a student stays consistent (0 to 0 or 1 to 1) for a rubric item, then we give a transition score of 0.


Discount Factor

When we have a transition score of +1 (i.e. when a student goes from 1 to 0 or from having a rubric item to not having that rubric item), we apply a discount factor of 0.9 for every time we have seen a transition from 1 to 0 after the first transition from 1 to 0 for this rubric item. For example, if a student goes from 1 to 0, then 0 to 1, then 1 to 0, the first transition score will be +1, the second transition score will be -0.5, and the third transition score will be +0.9.

For an individual problem, a studentâ€™s problem score is calculated by first taking the sum of the transition scores and then dividing by the square root of the number of submissions. This helps us to differentiate between students who may have spammed submissions and students who submit a reasonable number of submissions. We take the square root of the number of submissions to prevent the problem score from shrinking too quickly. To calculate a studentâ€™s total learning score, we sum up all of these problem scores.
